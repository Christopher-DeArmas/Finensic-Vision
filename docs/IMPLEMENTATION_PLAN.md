# Sentinel AML — Implementation Plan

The project is built in staged increments. Each stage produces something
runnable/testable before the next begins. This document is the master plan;
check items off as stages complete.

Legend: ✅ done · 🚧 in progress · ⬜ not started

---

## Stage 1 — Backend + Database ✅

**Goal:** a normalized SQLite database, ORM models, and a synthetic data
generator that seeds realistic data plus planted laundering scenarios.

- ✅ Directory scaffold (clean architecture)
- ✅ Docs: README, ARCHITECTURE, DATABASE_SCHEMA, this plan
- ✅ `core/config.py` — settings via env vars
- ✅ `core/database.py` — engine, session, `Base`
- ✅ `models/` — customers, accounts, merchants, transactions, risk_scores,
  alerts, cases, sar_reports, relationships + join tables
- ✅ `utils/` — enums, geo distance, id helpers
- ✅ `data/generator.py` — Faker-based generator (150 customers, accounts,
  merchants, thousands of transactions, 9 planted scenarios)
- ✅ `scripts/seed_db.py` — create tables + seed + print summary
- ✅ Verified: row counts and scenario spot-checks

**Deliverable:** `python -m scripts.seed_db` produces `sentinel.db`.

---

## Stage 2 — AML rule engine + risk scoring ✅

**Goal:** pure, explainable heuristics that score customers.

- ✅ `rules/base.py` — `Rule` protocol, `RuleResult(points, reason, severity, transactions)`
- ✅ Rules 1–10: structuring, rapid movement, circular transfers, dormant
  awakening, velocity, geographic anomaly, multiple incoming sources,
  high-risk jurisdiction, large cash deposit, account explosion
- ✅ `rules/engine.py` — run all rules for a customer, aggregate + normalize 0–100
- ✅ `services/scoring_service.py` — load context, run engine, persist `risk_scores`
- ✅ `services/alert_service.py` — raise alerts above threshold
- ✅ Unit tests per rule against the planted scenarios

## Conventions

- **Backend:** async where it matters, type hints everywhere, business logic in
  `services`, DB access only in `repositories`, rules stay pure.
- **Frontend:** TypeScript everywhere, reusable components, no duplicated logic.
- **Config:** all secrets/tunables via env vars; never hard-coded.
- **Reproducibility:** a fixed `SEED` yields identical data + scenarios.
